# 异常协议报文检测与防护系统 · 详细设计文档

**版本** v1.0　**定位** 通信设备(路由器/交换机)控制面协议栈内嵌部署　**适用协议** OSPF / IS-IS / BGP，及链路层报文处理模块(LDM)等

---

## 0. 阅读指引:这套系统是什么、不是什么

先把立场讲清楚,因为它决定了后面每一个设计取舍。

**它是一套"异常报文事后归因 + 免疫记忆"系统,不是一套"根因枚举 + 事前约束"系统。** 现网协议模块的崩溃、路由误删、状态机卡死,绝大多数源于**未知的实现缺陷**。缺陷是无穷无尽、逐个不同的,试图为每一类缺陷预置一条检查,永远追不上,也不可维护。本系统把协议实现当**黑箱**:平时不理解它,等它出问题之后,靠**关联(correlation)而非理解(comprehension)**,把"是哪个报文把它搞坏的"快速揪出来,记下报文特征,复发即拦。

**它检测的是"设备可见地坏掉"的形态,不是根因。** 根因不可枚举,但"设备坏掉"的可见形态是极少、稳定、且与协议无关的——崩溃、状态机卡死、批量删路由、报文风暴。装一个"批量删路由"探测器,能抓住**所有**以批量删路由形式发作的缺陷,不管底层根因是什么、我们知不知道。这与"给某个具体 bug 加规则"是方向相反的两件事。

**持久化特征库的首要价值,是打断死循环,不是"下次挡住"。** 崩溃 → 重启 → 从邻居重新学到毒化路由 → 再次崩溃,这才是"无法自愈、故障扩散"的真正形态。特征库必须在协议模块恢复与 GR 状态回放**之前**加载生效,才能真正止血。

**诚实的能力边界(不粉饰):** 首个受害者仍会中招(签名式防护的本质);纯竞态/时序缺陷提炼不出报文特征,只能靠自愈兜底;合法报文 + 内部缺陷的情形,系统会正确地判定"这不是异常报文问题",转自愈而非拦截。

## 1. 背景与设计目标

### 1.1 问题背景

协议模块因收到异常报文而异常重启、误删路由、内部状态机卡死,导致业务长时间中断且无法自愈,并可能使故障扩散全网。本设计以以下真实/构造案例为锚点驱动:

| 编号 | 案例 | 本质 |
|---|---|---|
| C1 | OSPF 收到 LSID=0.0.0.0 的 Network-LSA,域内 SPF 计算次数达特殊值时触发缺陷,误删域内路由 | 报文中的异常值触发实现缺陷 |
| C2 | EVPN 已协商扩展下一跳,主备倒换时 EVPN 备升主向 RM_IPv4 平滑订阅下一跳,按 IPv4 长度截断 IPv6 下一跳发起订阅,返回结果不匹配,升主状态机卡在下一跳订阅态,倒换流程无法继续,备板冗余失效 | 合法报文 + 内部截断缺陷,仅倒换触发 |
| C3 | LDM 解析目的为本机地址的 IPv6 IPsec 报文,命中管理面防护策略进入丢弃处理,丢弃后误释放整个报文内存,解析下一个报文时访问已释放内存,设备异常复位 | 合法报文命中丢弃路径,代码错误释放,下一个报文崩溃 |
| C4 | IS-IS 某设备时钟补偿变量内存跳变,协议计时器快 100 倍,始发 Purge;全网泛洪、反复删路由,IS-IS 路由全网震荡 | 本机内存损坏,合法 Purge 毒化全网 |

**范围外(本期不作为重点):** BGP 短期海量 UPDATE 导致重启、BGP 反复震荡导致 CPU 冲高。这两类是资源耗尽问题,靠水位/反压/max-prefix 的优雅降级解决,机制成熟且与本系统的报文归因主线正交,单独治理。

### 1.2 设计目标

对应任务书的三条目标:

1. **识别并拦截异常报文。** 但直接拦截并非主要目标——部分异常报文直接拦截会导致邻居断连、路由不同步。故默认处置是**非破坏性**的隔离,而非丢弃。
2. **异常事件发生后,识别并阻止问题扩散,进而自愈。** 覆盖批量路由老化、状态机卡死等。
3. **(主要目标)持久化记录异常报文特征,再次收到同类报文后拦截。** 核心价值在于打断"崩溃—重启—复发"死循环。

### 1.3 设计原则(核心立场)

- **黑箱归因,不枚举根因。** 不为每个缺陷预置检查;缺陷未知,靠事后关联定位。
- **检测发作形态,而非根因。** 触发器只覆盖崩溃/卡死/批量删/风暴这个极少而稳定的集合。
- **不重放。** 活体重放(再拉起协议实例)开销过大,也无法还原当时的计时器、内存布局、引用计数、竞态时序,已从设备侧机制中移除;归因改用溯源标签、驻留态扫描、跨事件关联。
- **不复制路由数据。** 不物化第二份 RIB/LSDB。持久化的只有紧凑指纹、对象上的小溯源标签、以及正比于字段数的分布画像。
- **稳态零开销。** 稳态只做 O(1)/对象 的轻量记录;归因等重活只在异常触发后的慢路径跑。
- **默认非破坏 + 人在环。** 系统产出"候选 + 置信度 + 证据链"交人晋级;仅高置信 + 非破坏处置的窄情形允许自动进观察态灰度。
- **本机学习本机生效,不自动向邻居扩散阻断规则。** 跨设备走现有可信配置分发通道 + 签名 + 灰度 + 可回滚。

### 1.4 术语

- **PDU IR(协议规范化字段树):** 报文在解析瞬间的规范化字段视图,仅存在于处理的那几微秒,处理完即丢弃,**不建并行数据库**。
- **溯源标签(input_id):** 挂在驻留协议对象上的 4–8B 引用,指回创建/最后修改该对象的接入事件。
- **黑匣子:** 复位后不清零的保留内存/NVRAM 中的环形缓冲,记录报文与状态变更。
- **绊线(trip-wire):** 通用异常触发器,一条覆盖一整类发作形态。
- **特征(signature):** 归一化字段上的谓词 + 处置等级,可持久化,复发即匹配。
- **驻留态:** 当前仍存在于 LSDB / Adj-RIB-In 等结构中的协议对象集合。

## 2. 总体架构

### 2.1 双轨结构

系统分两条轨道。**在线轨**是报文处理流水线,每个报文都走,稳态只跑廉价动作;**离线轨**是免疫闭环,只在出事时才被触发。

```
                          ┌──────────────── 在线轨(每报文) ────────────────┐
   报文入口 ──► 归一化IR ──► 不变式校验 ──► 特征库匹配 ──► 协议状态机 / LSDB / RIB
   (record_input)          (先天免疫)      (免疫记忆)       (report_event 上报状态变更)
        │                                     ▲                    │
        │ 写接入日志                          │ 加载/命中           │ 状态变更事件
        ▼                                     │                    ▼
   ┌────────────────────────── 黑匣子(复位可存活) ──────────────────────────┐
   │      报文指纹+元数据  ·  状态变更  ·  溯源标签   —— 同一条时间轴          │
   └────────────────────────────────────────────────────────────────────────┘
        ▲                                                            │
        │ 证据                                                       │ 触发
        │                          ┌──────────── 离线轨(仅出事时) ──┘
        │                          ▼
   健康监测(触发器) ──► 归因定位 ──► 特征生成 ──► 持久化特征库 ──► (人工审核后)分发
   崩溃/卡死/批量删/风暴   五级漏斗    泛化+置信度    掉电不丢         走可信配置通道
                                          │
                                          └──► 低置信/合法报文 → 自愈 + 上报,不出特征
```

在线轨对应目标 1;健康监测 + 自愈对应目标 2;黑匣子 + 归因 + 特征生成 + 持久化对应目标 3(主要目标)。

### 2.2 五级处理流程

离线轨的核心是五级流水线,全程把缺陷当未知:

1. **异常检测(触发器)** —— 通用绊线识别"设备坏掉"的形态。
2. **黑匣子(证据留存)** —— 复位可存活,报文与状态变更同一时间轴。
3. **归因定位** —— 漏斗式从"记的所有东西"收敛到 1–3 个候选报文 + 理由。
4. **特征生成与持久化** —— 异常模型一鱼两吃:排真凶 + 指出判别字段;泛化成最小谓词;掉电不丢。
5. **分级处置与自愈** —— 非破坏处置为默认;止损、隔离、自愈,阻止扩散。

### 2.3 部署形态与组件

- **捕获面(协议进程内):** 五个钩子(见 2.4)在报文热路径上由协议 inline 调用,**必须在进程内**——热路径不能引入 IPC。协议团队实现少量适配(IR 映射、状态上报点)。
- **分析面 / PAD 服务(控制面独立进程):** 承载触发器汇聚、归因引擎、特征库管理、异常模型;做成独立进程是为**故障隔离**——协议崩溃后,一个全新的 PAD 仍能读回证据做事后归因。与协议进程的数据交换机制见 2.5。
- **持久化存储:** Flash/NVRAM,双份 + 校验 + 版本。
- **线卡执行插件:** 纯头部可判的特征下压到 NP/ACL/CoPP,零 CPU 执行。
- **网管侧分析器:** 集中训练异常模型、汇总候选、承载人工审核与灰度分发。

### 2.4 协议模块钩子 API

协议团队只需接入这五个钩子,其余由框架承担:

```c
// 报文入口:查特征库 + 不变式,返回处置(观察/隔离/丢弃…)
disposition_t ingress_check(const pdu_ir_t *ir);

// 写接入日志(环形、复位可存活),返回溯源ID
input_id_t    record_input(const pdu_t *pdu, const meta_t *meta);

// 给协议对象(LSA / 属性集 / 下一跳)打溯源标签
void          tag_object(object_t *obj, input_id_t id);

// 上报状态变更:LSDB/RIB 增删、FSM 迁移(尽量带上引起它的 input_id)
void          report_event(const state_event_t *ev);

// 特征库查询(供快/慢路径复用)
const feature_t *query_signature(const pdu_ir_t *ir);
```

### 2.5 数据获取与进程边界

分析面做成独立进程,立刻带出一个问题:**PAD 怎么拿到协议组件的数据?** 关键是把捕获面与分析面分开,数据靠**协议往外 push,不是 PAD 往里 pull**。

```
  协议进程(捕获面·进程内)                    PAD 进程(分析面·独立)
  ┌──────────────────────────┐             ┌──────────────────────────┐
  │ 报文热路径 · 5 钩子        │  ③ 服务请求 │ 归因引擎 / 异常模型       │
  │ LSDB / RIB / FSM          │ ◄─────────► │ 特征库管理               │
  │ 慢路径 worker 线程        │ (计算下沉)  │ (仅回传 top-K 候选)      │
  └───────────┬──────────────┘             └─────────────┬────────────┘
              │ ① push 写入(热路径)                       │ ② 读取
              ▼                                           ▼
  ┌───────────────────── 共享复位可存活区(共享内存 / NVRAM)─────────────────┐
  │ 黑匣子:接入日志 + 状态变更日志(有界时间窗) · 特征库 —— 协议崩溃后仍可读  │
  └──────────────────────────────────────────────────────────────────────────┘
  既有流 BMP / BGP-LS ──(流式训练模型,不落副本)──► PAD
```

PAD 拿数据的三条路:

| 路径 | 承载数据 | 是否落副本 | 触发时机 | 用于 |
|---|---|---|---|---|
| ① 钩子 push 到共享区 | 报文指纹、状态变更、溯源标签 | 否(有界时间窗,非路由镜像) | 热路径 | 全部;卡死溯源(C2) |
| ② 既有跨组件流(BMP/BGP-LS) | Adj-RIB-In / LSDB 流 | 否(流式训练模型即弃) | 稳态 | 训练异常模型 |
| ③ 服务请求(计算下沉) | 仅 top-K 候选 + 字段 | 否(不拷贝 LSDB) | 慢路径,仅协议存活 | 驻留态扫描(C1) |

**① 钩子 push(热路径,无 IPC)。** `record_input / tag_object / report_event` 只是往共享内存的单生产者无锁环写几个字节,协议不阻塞、不等 PAD。一个关键设计:`report_event` 上报 FSM 卡死时**把卡住对象的 `input_id` 一并带出**,PAD 在共享日志里直接 `input_id → 接入条目`做溯源追溯,**根本不读协议的私有内存**(C2 走这条)。

**② 复用既有跨组件流。** componentized 的栈(VRP 亦然)里,协议与 RM 本就分开、本就通过既定接口交换路由:BGP 的 Adj-RIB-In 用 **BMP(RFC 7854)** 导流、链路状态用 **BGP-LS(RFC 7752)/LSDB 导出**。这些数据本就跨进程边界,PAD 作为又一个消费者订阅即可,无需新的 reach-in。**注意:这条流只用来流式更新异常模型的 sketch,消费一条更新一条即弃,不落地成影子表。**

**③ 服务请求:把计算下沉到数据(慢路径,仅协议存活时)。** 唯一真正需要读活体状态的是**驻留态异常扫描**(C1)。此时 PAD 不去够协议的私有堆,而是发请求,协议在**一个非阻塞 worker 线程**(不是卡死的那个 FSM 线程)上遍历自己的 LSDB、对每个对象套 PAD 给的打分函数、**只回传 top-K 候选 + 字段值**。计算搬到数据那边、数据不过界,协议自己 walk、自己上锁,一致性由它保证。

**按故障态分流(独立进程的真正价值):** 协议**活着**时(C1 误算、C2 卡死),活体状态可得——C1 用服务请求扫 LSDB,C2 的卡死线程被阻塞但协议**其他线程**照常响应查询;协议**死了**时(C3 崩溃),活体状态根本不存在也不需要,归因由**一个全新的 PAD** 读回共享区的报文时间轴、跨事件取交集完成。这正是共享区必须复位可存活、且分析面必须独立于协议进程的全部理由:协议连同私有堆都没了,证据还在。

#### 2.5.1 内存开销:明确不引入第二份路由数据

本设计的核心取舍是**坚决不为归因维护第二份路由数据(影子 RIB)**——这是响应"多复制一份数据的开销"这一顾虑的关键。

**常驻内存只有三样,没有一样是路由的拷贝:**

- **溯源标签:** 4–8B × 对象数,且打在**引用计数共享的属性集**上(属性集数量远小于路由数,靠共享去重),MB 级。
- **黑匣子环形区:** 64–256MB,是一个**有界时间窗**的报文指纹 + 事件,不是路由集合的镜像;大小由窗口 × 速率封顶,**与路由规模解耦**。
- **字段分布模型:** 按字段的聚合 sketch(count-min / 直方图),大小正比于**字段数**、与对象数无关,亚 MB 级。

**两条"消费流而非存副本"的纪律:** 既有跨组件流(BMP/BGP-LS)只流式训练模型、即用即弃,不建影子表;驻留态扫描把打分函数下沉到协议 worker、只回传 top-K 候选(KB 级),不拷贝 LSDB。二者都不占额外的路由内存。

**反证——若改为维护影子副本供解耦扫描,代价是多少:** 影子 Adj-RIB-In = (指纹 + 打分字段) × 路由数,约 32B/路由 × 百万路由 ≈ 32MB、× 5 百万路由 ≈ 160MB,且**随路由规模线性增长**。本设计用"计算下沉"避开了这笔与路由规模挂钩的开销。之所以能避开,是因为**驻留态扫描只在协议存活时才需要**(误算类);崩溃类归因走黑匣子报文时间轴、不做驻留扫描,因此不存在"崩溃后仍需扫驻留态、从而必须常驻影子"的场景。

**一致性:** 共享环形区单生产者(协议写)单消费者(PAD 读)无锁;特征库 PAD 偶尔写、协议热路径频繁读,用版本化指针切换(RCU 式),读侧永不阻塞。

## 3. 异常检测(触发器)

### 3.1 设计立场

**"根因"无穷无尽、逐缺陷不同,不可枚举;但"设备可见地坏掉"的方式极少、稳定、与协议无关。** 触发器只盯后者。这是本系统不退化成"见一个 bug 加一条规则"的第一道保证:每个触发器覆盖一整类发作形态,而非一个具体缺陷。

### 3.2 四类通用触发器

**(1) 崩溃 / 复位。** 最直接,但要求黑匣子复位可存活(见第 4 章)。复位后读回黑匣子做归因。覆盖 C3。

**(2) 状态机看门狗。** 所有 FSM(BGP FSM、OSPF NSM/ISM、下一跳迭代、RIB→FIB 下发、主备批量同步)统一框架注册,强制携带 `进入时间 / 最大驻留时长 / 进展计数器`。驻留超时或进展计数不前进 = 卡死。零定制逻辑,一个框架吃下所有状态机卡死。覆盖 C2(卡在下一跳订阅态)。

**(3) 批量路由变更 + 因果绊线。** 窗口内路由增删量过阈值即触发。更关键的是**因果一致性绊线**:若 ΔLSDB≈0、无接口/邻居/配置事件,却出现 ΔRIB 巨大(如 −4.8 万),则判定"是本机自己算错了"。这是最便宜也最有效的一条——不需要知道是哪个缺陷,只需知道"大结果没有相称的原因 = 可疑"。覆盖 C1。

**(4) 按对象的速率 / 重复模式。** 按协议对象(某条 LSP、某个前缀)做令牌桶 / EWMA,识别"同一对象被反复 purge+重生 / 反复震荡"的异常速率。覆盖 C4(purge 风暴)。合法拓扑变更引发的 purge 是有界的,风暴是"同一 LSP 高频反复",据此把两者分开。

### 3.3 性能约束

触发器的核心纪律:**监控开销正比于异常率,而非流量率。** 稳态只跑这四个绊线,全是 O(1) 且尽量不在报文热路径——看门狗放低优先级线程周期扫,水位/速率就是本来要加的计数器,因果绊线只在有删除动作时比对两个已有的计数。昂贵的东西(归因、驻留扫描、影子校验)只在绊线触发后跑。

## 4. 黑匣子(证据留存)

### 4.1 复位可存活要求

**这是 C3 教给我们的硬约束:硬 fault 导致复位时,根本没有机会跑落盘逻辑。** "崩溃前先落盘"只对可捕获的软异常成立。故黑匣子必须**连续写入一块复位后不清零的保留内存区(或 NVRAM),复位重启后再读回**。归因引擎(在独立的 PAD 进程)因此能在协议模块崩溃、重启的同时照常取证。

### 4.2 双日志与统一时间轴

黑匣子记两条日志,放在**同一条时间轴**上,这是"批量删 4.8 万能被关联、因果绊线能算 Δ因 vs Δ果、崩溃能跨事件对齐"的底座:

- **接入日志(ingress journal):** 追加写、有界环形。每条含 `时间戳 / 入接口 / 邻居 / 协议类型 / 报文指纹 / 关键字段紧凑拷贝`(新奇或高危报文存全量),按 `input_id` 索引,使溯源标签能解析回具体条目。仍被驻留对象引用的条目**钉住不老化**(至少保留其关键字段摘要),保证活对象的溯源永不悬空。
- **状态变更日志(state journal):** LSDB/RIB 增删、FSM 迁移,每条尽量带上引起它的 `input_id`。

### 4.3 自适应保真度

稳态只存指纹/摘要;风险指标(命中率、震荡率、内存增长率)升高时提升保真度;异常/新奇报文**永远全量**。这样既控内存,又保证真凶来临时能抓到全字节。若首次异常时真凶只留了粗指纹,崩溃循环会补——第一次异常后抬高保真度,复发即抓全量(见 5.6)。

### 4.4 数据结构与内存核算

| 结构 | 内容 | 规模 | 说明 |
|---|---|---|---|
| 溯源标签 | 4–8B input_id | O(对象数),每个极小 | 打在本就引用计数共享的属性集上,近乎零成本 |
| 接入日志 | 指纹+关键字段+全量(高危) | 有界环形(窗口×速率,封顶) | 复位可存活 |
| 状态变更日志 | 增删/迁移事件 | 有界环形 | 同一时间轴 |
| 字段分布模型 | 按协议按字段的 sketch | 正比于**字段数**,与对象数无关 | count-min + 小直方图 + 均值方差 |
| 事件快照区 | 触发时的支点/窗口/工作集 | 单次事件级 | 让归因在模块重启时仍能跑 |

**结论:全程没有第二份路由数据。** 标签搭现有结构便车,日志是封顶环形,模型正比于字段数。

## 5. 归因定位

归因是整个系统里唯一"难"的部分,别的都是工程。它不是一个算法,而是一个**从"黑匣子里记的所有东西"漏斗式收敛到"1–3 个候选报文 + 为什么是它们"的过程**,全程不重放。

```
   异常触发 ──► ①入口分类(支点+窗口) ──► ②三路并行缩小 ──► ③打分融合 ──► ④判别字段 ──► ⑤出口闸门
                                          ├ 溯源链追溯                                    ├► 出特征
                                          ├ 驻留态异常扫描                                └► 内部bug:自愈+上报
                                          └ 跨事件时间关联
```

### 5.1 入口分类

触发器一响,第一件事不是找报文,而是把异常翻译成一次具体查询:一个**支点(pivot)** + 一个**搜索窗口**。这决定下游走哪条路,而这依然是对"发作形态"分类,不是对根因分类。

| 触发形态 | 支点 | 走向 |
|---|---|---|
| 卡死 | FSM 卡住的那个对象(订阅的下一跳/路由/邻居) | 溯源链追溯 |
| 批量删 + 因果绊线 | 被删前缀集;但 ΔLSDB≈0 说明是计算错 | 驻留态异常扫描 |
| 崩溃复位 | 时间游标(故障前最后若干次操作) | 跨事件时间关联 |
| 报文风暴 | 正在风暴的对象类 + 速率模式 | 速率/重复模式抑制 |

### 5.2 三路并行缩小器

三种互相独立的缩小器,由入口分类决定跑哪几路,各吐候选 + 理由。**全部是通用机制,不含逐缺陷知识。**

**(A) 溯源链追溯** —— 用于支点指向具体对象的情形。顺着对象上的溯源标签:卡死的下一跳 → 路由 → 属性集 → 那条 Update → 邻居,O(1) 指针追溯,精度最高。要求对象可打标、发作点指向对象。C2 走这个。

**(B) 驻留态异常扫描** —— 对付时间解耦的关键一招,最重要的缩小器。**要点:真凶报文往往几小时前就来了,发作时早已滚出黑匣子窗口——但它变成的那个对象还躺在 LSDB 里。所以不翻历史报文流,去扫当前驻留状态,就地扫毒。** 拿设备自己历史训练的字段分布模型给当前 LSDB 每条 LSA 打分,LSID=0 当场点亮。O(N) 一次、只在触发后跑。C1 走这个。

**(C) 跨事件时间关联** —— 用于崩溃这种"什么都没留下"的情形。**破法是利用崩溃循环本身**:崩溃→重启→复发给了一批重复样本。取"每次崩溃前都出现的那个报文"的**交集**——受害报文每次都在变(碰巧排在后面的那个),真凶每次都不变。交集自动滤掉变量、留下不变量。**让缺陷变坏的那个"循环",恰恰是让它可归因的东西。** C3 走这个(真凶是命中丢弃路径、错误 free 的那个 IPsec 报文,崩溃点在它后面的无辜报文,交集把无辜报文滤掉)。

### 5.3 打分融合

三路可能同时出候选,需融成一个带置信度的排序。这里就是所谓"小模型",但它是**透明的加权打分,不是黑盒神经网络**——因为证据链必须能被运维看懂、看懂了才敢晋级成拦截,**可解释性比准确率更重要**。每个候选算以下信号加权:

- **异常分:** 字段取值相对本机基线有多罕见(isolation forest / one-class SVM / 按字段经验分布 z-score)。
- **溯源强度:** 是否被标签直连到发作对象。
- **复现度:** 是否每次崩溃前都出现。
- **因果邻近:** 是否落在算错的那块区域里。
- **新奇度:** count-min sketch 说它历史计数≈0。
- **触发条件吻合:** 进入时间是否与 SPF 计数轨迹等特定条件对得上。

输出是排序候选 + 一个**标定过、诚实的**置信度。**没有一个候选分高时,输出就是"低置信,疑似内部缺陷,需人工"**,这正是 C2 的结局。

### 5.4 判别字段提取

对头号候选,不止说"是这个报文",还要说"是它的哪几个字段异常"。**上一步的逐字段异常分直接给出这个:值本身异常的字段(ls_id=0)就是判别字段,进谓词;天然变化的字段(LS Age、序列号、校验和、源 IP/MAC)按构造掩码掉。** 这就是砍掉重放后、替代"逐字段消融重跑"的机制——异常模型既排真凶,又指出该拿哪几个字段做签名,得到最小谓词 `type==2 && ls_id==0`。

**边界:** 单字段异常和简单跨字段关系可自动化;复杂语义条件(某字段单看正常、结合另一字段才异常且逻辑复杂)自动模型抓不全,交人工收窄。

### 5.5 出口闸门(二维判据)

"要不要出特征"不是只看异常分,而是**二维的:候选异常度 × 处置安全性**。满足其一即可出特征:

- **候选明确异常**(如 LSID=0),或
- **处置可证明非破坏**(即拦截结果与本来就会发生的一致)。

| 案例 | 分类→模式 | 起效缩小器 | 融合主信号 | 二维闸门 → 产物 |
|---|---|---|---|---|
| C1 OSPF LSID=0 | 批量删+绊线 → 驻留扫描 | 驻留态异常扫描 | 异常分极高+新奇 | 异常高 → 出值特征 |
| C3 LDM UAF | 崩溃 → 时间游标 | 跨事件时间关联(交集) | 复现度极高 | 异常中,但处置=提前丢本就要丢的 → **安全,出特征** |
| C2 EVPN 卡死 | 卡死 → 对象支点 | 溯源链追溯 | 溯源强、异常分低 | 合法+处置会误杀 → **不出特征,自愈+上报** |
| C4 IS-IS purge | 风暴 → 速率 | 速率/重复模式 | 行为异常 | 行为特征 → 抑制该 LSP 的 purge 收发 |

关键对照:C3 候选异常分只是中等却仍出特征,**因为处置是"把本来就要丢的报文提前到 CoPP 丢",结果完全一致、误杀为零**;C2 候选被溯源链直连锁定,但异常分低(合法路由)、且拦它会误杀所有合法 EVPN 路由 → 闸门判定"这不是异常报文问题,是内部缺陷"。**没有任何一条针对 C2 缺陷的规则,是逐字段扫描找不到异常 + 处置不安全,共同把它推向自愈通道的。**

### 5.6 时间解耦与崩溃循环的处理(小结)

两条最难的路径,处理方式已固化:

- **时间解耦(真凶几小时前进来):** 用驻留态扫描——毒还在对象里,扫对象不扫历史。
- **崩溃无留存:** 用崩溃循环——重复样本取交集,真凶不变、受害多变。第一次异常后抬高保真度,复发抓全量。**首个受害者可能只得到粗候选,复发才拿到精确特征**,这与"首受害者中招、之后免疫"一致。

## 6. 特征生成与持久化

### 6.1 特征结构

```
Feature {
  feature_id       // 唯一ID
  protocol         // OSPF / ISIS / BGP / LDM ...
  match_predicate  // 归一化字段谓词,如: ospf.lsa.type==2 && ls_id==0.0.0.0
                   //                     afi_safi==EVPN && purge_rate(lsp_id) > θ
  scope            // 本邻居 / 本区域 / 本AFI-SAFI / 全局
  disposition      // 观察 / 净化 / 隔离 / 降级 / 丢弃(见第7章阶梯)
  confidence       // 归因置信度 [0,1]
  state            // observe(灰度) / enforcing(生效) / suspended(挂起)
  hit_count        // 命中计数
  first_seen, last_seen
  evidence_ptr     // 指向黑匣子事件 / incident dossier
  source           // local-learned / nms-pushed / vendor-published
  signature        // 来源签名(外部下发必须)
  ttl              // 老化时间(可选)
}
```

### 6.2 泛化策略

特征必须建立在**协议语义对象**上,不能是报文字节——同一条毒化 LSA 会被反复泛洪,每次的 IP/MAC 头、LS Age、序列号、校验和都不同,字节哈希永远匹配不上第二次。归一化后,判别字段来自 5.4 的逐字段异常分,天然变化字段一律掩码。这样得到的是最小且精确的谓词,不靠拍脑袋写正则,也不靠重放。

### 6.3 持久化与加载时机

- 落 Flash/NVRAM,双份 + 校验 + 版本号。
- **加载时机必须早于协议模块恢复与 GR 状态回放。** 否则毒化对象会从 checkpoint 里"复活",死循环打不断。这是特征库能真正止血的前提。

### 6.4 驻留态隔离扫描

加载后,对已驻留状态(LSDB / Adj-RIB-In)按特征库扫描一遍并隔离,否则毒化对象会从 checkpoint 复活。C1 复发时:重启后加载 `type==2 && ls_id==0` → 扫 LSDB → 该 LSA 隔离(继续泛洪保证 LSDB 一致,但不参与 SPF)→ 死循环打断。

## 7. 分级处置与自愈

### 7.1 分级处置阶梯

处置必须分级,不能只有"丢/不丢"——直接拦截可能导致邻居断连、路由不同步。默认走非破坏级别。BGP 侧可直接复用 RFC 7606 的现成阶梯。

| 级别 | 动作 | 破坏性 | 适用 |
|---|---|---|---|
| 观察 | 只记录不干预 | 无 | 新特征灰度期 |
| 净化 | 属性丢弃 / 字段裁剪 | 低 | RFC 7606 attribute-discard;裁剪坏字段 |
| 隔离 | treat-as-withdraw / **入库但不参与计算** | 低 | 毒化 LSA、毒化路由。链路状态类靠"继续泛洪但本机不算路"保证 LSDB 一致 |
| 降级 | 关该 AFI/SAFI / 降优先级 / 停止引入 | 中 | 反复出问题的能力 |
| 丢弃 / 复位 | 丢包 / 复位会话 | 高 | 明确恶意、无其他手段时 |
| 设备隔离 | overload bit / max-metric / GRACEFUL_SHUTDOWN 引流 | 高 | 本机判定不可信,把流量引走 |

对 C1 这种链路状态类,"**入库 + 继续泛洪但不参与 SPF**"是最关键的一招:LSDB 一致性靠继续泛洪保证(否则邻居 DD 交互卡在 Loading,全域抖动),而本机拒绝拿它算路。这正好解掉"直接拦截会导致邻居断连、路由不同步"。

### 7.2 硬红线

**学习出来的特征永远不允许拦截 Hello / IIH / KEEPALIVE / OPEN。** 否则防护系统自己会制造邻接断连——比它要防的缺陷更糟。这条无条件优先于任何特征。

### 7.3 自愈与止损机制(目标 2)

这些是通用机制,不是逐缺陷补丁:

- **状态机看门狗 + 回退:** 卡死时执行安全回退。C2 的自愈动作是**放弃平滑订阅、回退到全量冷订阅**——慢,但让倒换走完,避免"备板冗余永久失效"。
- **路由删除熔断器:** 批量删触发后冻结——不下发 FIB 删除、不向邻居传播撤销、不生成汇总/引入,同时置 max-metric 引流。
- **结果校验器(怀疑时才跑):** 熔断后用独立实现的"影子 SPF"(简单、经充分验证的参考实现,跑在备板空闲核,或经 BGP-LS 把 LSDB 送控制器算)重算比对。不一致 → 判定主实现算错 → 拒绝应用这次删除,在 GR/NSR 保护下重启计算。这是 N-version programming 的经典用法。C1 用它当场止血。
- **节点自我隔离(通用原则):** 检测到本机内部一致性被破坏时,**fail-stop 而非 fail-noisy**——停止始发、置 overload、告警,不带着损坏状态继续参与协议。宁可自己停,不要把故障扩散全网。

**关于 C4(purge 风暴)的处置定位:目标不是检测源端时钟不一致,而是杜绝全网收发 purge。** 落地即——异常触发器发现"某 LSP 被反复 purge+重生、速率远超正常",判为风暴;处置是**每台设备本地抑制该 LSP 的 purge 传播与接收并 hold-down**。每台设备独立检测、独立抑制,风暴就在报文层被就地掐断、扩不出去,**根本不需要知道源头那台的时钟缺陷**。这是把行为异常作为"异常报文模式"来遏制的典型。

### 7.4 倒换前健康闸门

C2 的二次故障是"备板同步没完成 → 冗余失效"。必须在倒换前校验备板:同步 FSM 是否完成、RIB 摘要哈希是否与主板一致。不健康就**禁止倒换并告警**,而不是盲目倒换把故障扩散到备板。这是一条通用的"不切到不健康的备板"闸门,不是针对某个缺陷。

## 8. 特征分发

**自动向邻居扩散阻断规则本身就是新的攻击面和故障面,一旦判错影响极大,本系统明确不做。**

- **本机学习、本机生效。** 特征只在本地以观察/非破坏方式学习与生效。
- **上送与人工审核。** 候选特征连同证据上送网管,由人(或受控变更流程)审核。
- **分发通道与安全。** 批准后走**现有可信配置分发通道**(NETCONF/gRPC/控制器)下发——与任何一次策略下发同一套信任模型,不做设备间"八卦式"P2P 传播。分发必须**签名、灰度(先金丝雀几台、观察态、看指标、再扩)、可回滚**;设备**拒收未签名特征**;批准后仍默认非破坏处置,除非显式升级。

这样,一台设备当哨兵,经人工确认后全网受控免疫,而不是让一条可能错误的规则自动黑洞掉一整段前缀。

## 9. 典型案例端到端走查

### 9.1 C1 · OSPF LSID=0 批量删路由(出值特征)

致病 LSA 进入 LSDB → **因果绊线**发现"ΔLSDB≈0 但删了 4.8 万路由" → 熔断,冻结 FIB 删除、停止传播撤销、置 max-metric 引流 → **入口分类**判为计算错,走**驻留态异常扫描** → 扫当前 LSDB,LSID=0 那条异常分极高 → **影子 SPF** 重算比对,确认主实现算错,拒绝应用删除、GR 保护下重启 SPF → **判别字段**提取 `type==2 && ls_id==0`,age/seq/checksum 掩码 → **闸门**:高异常 + 非破坏处置(隔离出 SPF),出特征、观察态灰度 → 人工确认后生效。复发时加载即隔离,泛洪照旧、SPF 不采纳,**死循环打断**。

### 9.2 C3 · LDM IPv6 IPsec UAF(出值特征,处置安全)

报文命中管理面丢弃策略,丢弃路径错误 free 整个缓冲,解析下一个报文时 UAF → 设备复位 → **复位后读回黑匣子** → **入口分类**为崩溃、时间游标 → **跨事件时间关联**:利用崩溃循环,取每次崩溃前的公共报文交集,滤掉多变的无辜受害报文,锁定命中丢弃路径的那个 IPv6 IPsec 报文 → **闸门**:异常分中等,但处置=**把本就要丢的报文提前到硬件 CoPP 丢**,结果一致、误杀零 → 出特征,绕开有缺陷的 free 路径,作为修好前的止损。**依赖黑匣子复位可存活**。

### 9.3 C4 · IS-IS purge 风暴(行为特征,遏制)

源端时钟跳变始发 purge → 各受害设备的**速率/重复绊线**发现"某 LSP 反复 purge+重生、速率超阈值" → 判为风暴 → **本地抑制该 LSP 的 purge 传播与接收并 hold-down** → 每台设备独立检测、独立抑制,风暴就地掐断、扩不出去。**不需要定位源端时钟缺陷**,直接杜绝全网收发 purge。源端若自身检测到内部一致性破坏,按 7.3 自我隔离(停始发、置 overload)是可选的二级防线。

### 9.4 C2 · EVPN 倒换卡死(正确地不出特征)

主备倒换,EVPN 备升主向 RM_IPv4 平滑订阅下一跳,按 IPv4 长度截断 IPv6 下一跳,返回不匹配,升主 FSM 卡在下一跳订阅态 → **看门狗**报卡死 → **入口分类**为卡死、支点=卡住的下一跳对象 → **溯源链追溯**:下一跳 → 路由 → 属性集 → 那条 Update,锁定一条**带 IPv6 下一跳的合法 EVPN 路由** → **判别字段**扫描找不到任何异常字段(它完全合法)→ **闸门**:异常分低 + 拦它会误杀所有合法 EVPN 路由 → 判定"这是内部缺陷,不是异常报文",**不出特征** → 转**自愈**:看门狗触发**冷订阅回退**让倒换走完;**倒换前健康闸门**在下次倒换前校验备板同步与 RIB 摘要,不健康则禁止倒换 → 上报人工/厂商修复内部截断缺陷。

**这一例证明系统不是枚举:它不认识这个缺陷,只是发现"归因出来的报文统计上正常",据此自动判定报文特征这条路不适用。**

## 10. 性能与资源开销

**稳态开销由两件廉价事构成,别无其他:** 接入时盖溯源标签(存几字节),和字段模型更新(几次 sketch 自增),都是 O(1)/对象,不扫描、不做模型推理。特征库匹配用两级(Bloom/哈希预筛 → 命中才完整谓词求值),稳态几乎全 miss,约 50–100ns/PDU;纯头部可判的规则下压 NP/ACL/CoPP,零 CPU。

**归因的贵活——驻留态 O(N) 扫描、融合、模型查询、影子校验——只在异常触发后的慢路径跑,稳态零开销**,且可放到备板空闲核或控制器上。这直接满足"不能降协议性能"的要求。

**内存:** 全程不复制路由数据(见 4.4)。逐路由的入口检查必须**嵌进现有解析循环**,复用解析时本就要访问的字段(cache 命中),不另起一遍遍历;OSPF/IS-IS PDU 速率本就低,BGP 收敛期速率高故逐路由检查尤其要就地做。溯源标签打在共享属性集上,百万路由量级是 MB 级;黑匣子占 64–256MB 复位可存活共享区。

## 11. 风险与规避

### 11.1 头号风险:误杀

**一条错误特征 = 自己制造大面积断路,比原来的缺陷更糟。** 反制必须内建:

- 默认处置非破坏(隔离/不参与计算/treat-as-withdraw,而非丢包复位)。
- 新特征强制**观察态灰度 N 小时**,只记录不拦截。
- 影响面预估超阈值(> X% 路由或 > Y 个邻居)**自动挂起**等人工确认。
- **自监控自动回滚:** 启用某特征后路由规模骤降则自动回滚该特征。
- **一键逃生开关:** 关闭所有学习特征。
- 外部下发特征必须签名;设备拒收未签名。

### 11.2 固有局限(诚实面对,不粉饰)

- **首个受害者仍会中招**——签名式防护的本质,靠受控分发把损失限制在一台设备。
- **纯竞态/时序缺陷提炼不出报文特征**——只能靠看门狗、熔断、自愈兜底。
- **合法报文 + 内部缺陷**——系统正确地归到自愈而非拦截(C2),不强行造特征。
- **黑匣子有界**——真凶全字节可能没抓到;驻留态类靠扫对象不受影响,崩溃类靠循环复发补抓。
- **复杂跨字段语义条件**——自动模型抓不全,交人工收窄。

## 14. 依赖与复用的标准

尽量复用现成标准,减少自研:

- **RFC 7606** —— BGP 分级错误处理,直接就是第 7 章的处置阶梯。
- **RFC 8950 / 5549** —— Extended Next Hop Encoding,C2 相关的下一跳编码基准。
- **GR(RFC 3623/5306/4724)与 NSR** —— 熔断后重启计算、倒换的保护伞。
- **RFC 6987(OSPF max-metric)、IS-IS Overload bit** —— 引流与自我隔离。
- **RFC 8326(GRACEFUL_SHUTDOWN)** —— 设备隔离时把流量优雅引走。
- **BMP(RFC 7854)** —— 导出 pre-policy Adj-RIB-In,天然是"抗原流"来源,可用于网管侧集中训练异常模型。
- **BGP-LS(RFC 7752)** —— 把 LSDB 导给控制器跑影子 SPF 做结果校验。

## 附录 A · 一句话总结价值主张

这套系统里,签名式防护(主要目标)只对少数纯"异常值"类缺陷完整有效;真正撑起"识别、止血、自愈、防扩散"的是四样通用能力:**通用触发绊线、复位可存活的黑匣子取证、漏斗式黑箱归因、以及分级非破坏处置与自愈**。特征库的最大价值不在"下次挡住报文",而在**打断"崩溃→重启→重新学到毒丸→再崩"的死循环**。首个受害者仍会中招、纯竞态缺陷提炼不出特征——这两条本质局限不加掩饰,靠自愈与源端自隔离兜底。
